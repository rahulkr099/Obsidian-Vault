This route handles **uploading a file to your server’s local disk**.  
Let’s walk through **how data moves step-by-step** from the browser → server → disk → memory metadata.

---

# 1. Big Idea

A user uploads a file using **form-data**.  
Your backend:

1️⃣ Receives the request  
2️⃣ Multer processes the file  
3️⃣ Saves the file on disk  
4️⃣ Creates metadata  
5️⃣ Stores metadata in memory  
6️⃣ Sends response to client

---

# 2. Pseudo Code (Route Logic)

```text
START

Client sends POST request to /upload/local
with form-data file

Multer middleware runs
    validate file type
    validate file size
    save file to UPLOAD_DIR

IF file upload failed
    return error response

Generate unique file id

Create metadata object
    id
    storage type
    original filename
    mime type
    size
    file path
    upload timestamp

Store metadata in uploadedFiles array

Return success response with metadata

END
```

---

# 3. Step-by-Step Data Flow

## Step 1 — Client Sends File

Example request from frontend:

```
POST /upload/local
Content-Type: multipart/form-data
```

Form data:

```
file = profile.png
```

Flow begins:

```
Browser
   ↓
Express Server
```

---

# 4. Step 2 — Multer Middleware Runs

Your route uses:

```javascript
uploadLocal.single('file')
```

Meaning:

```
Expect ONE file
field name = "file"
```

Multer now:

1️⃣ Reads the incoming file  
2️⃣ Checks **fileFilter**  
3️⃣ Checks **file size**  
4️⃣ Saves file to disk

---

### Example file object created by Multer

After upload:

```javascript
req.file = {
  originalname: "profile.png",
  mimetype: "image/png",
  size: 203423,
  path: "uploads/1712837474-abcd1234.png"
}
```

---

# 5. Step 3 — File Validation Check

Your code checks:

```javascript
if (!req.file)
```

### Pseudo Code

```
IF file missing OR rejected
    return error
```

Example response:

```json
{
 "message": "No file uploaded or invalid file type"
}
```

---

# 6. Step 4 — Generate Unique File ID

```javascript
const id = generateId();
```

Example:

```
generateId()

→ "d1a9c82f9a2e8bfa0c34ab22"
```

This ID will identify the uploaded file.

---

# 7. Step 5 — Create Metadata Object

Server creates:

```javascript
const meta = {
 id: "d1a9c82f9a2e8bfa0c34ab22",
 storage: "local",
 originalName: "profile.png",
 mimeType: "image/png",
 size: 203423,
 path: "uploads/1712837474-abcd1234.png",
 uploadedAt: "2026-03-09T22:20:00"
}
```

### Pseudo Code

```
metadata = {
    id
    storage type
    original file name
    mime type
    file size
    saved file path
    upload timestamp
}
```

---

# 8. Step 6 — Store Metadata in Memory

```javascript
uploadedFiles.push(meta)
```

Now your in-memory database becomes:

```javascript
uploadedFiles = [
 {
   id: "d1a9c82f9a2e8bfa0c34ab22",
   storage: "local",
   originalName: "profile.png",
   mimeType: "image/png",
   size: 203423,
   path: "uploads/1712837474-abcd1234.png",
   uploadedAt: "2026-03-09T22:20:00"
 }
]
```

# 13. How This Would Look in Production

Instead of memory:

```
uploadedFiles[]
```

You would store metadata in a **database**.

Example:

```
MongoDB
PostgreSQL
Redis
```

Typical production architecture:

```
User Upload
     ↓
API Server
     ↓
Multer
     ↓
Local Disk / S3
     ↓
Metadata → Database
```
