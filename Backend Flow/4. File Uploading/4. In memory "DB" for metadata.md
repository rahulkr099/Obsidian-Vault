This part of your code creates a **temporary database in memory** to store information about uploaded files. Think of it as a **small notebook inside your server** that remembers details about files.

We’ll understand it through:

- Simple idea
    
- Pseudo code
    
- Real request example
    
- How data moves inside the server
    

---

# 1. Big Idea

When a user uploads a file, the server usually saves **two things**:

1️⃣ The **actual file** (disk / S3)  
2️⃣ The **metadata** (file information)

Metadata includes things like:

```text
file id
original file name
mime type
file size
where it is stored
upload time
```

Your code stores this metadata in:

```javascript
const uploadedFiles = [];
```

This is just a **temporary array database**.

        upload time

# 3. The In-Memory Database

```javascript
const uploadedFiles = [];
```

### Pseudo Code

```text
uploadedFiles = empty list
```

Example stored data later:

```text
uploadedFiles = [
 {
  id: "a8c92f10",
  originalName: "resume.pdf",
  mimeType: "application/pdf",
  size: 102400,
  storage: "local",
  pathOrKey: "uploads/171234-abcd.pdf",
  uploadedAt: "2026-03-09T21:00:00"

```

Important thing:

⚠️ This **disappears when the server restarts**.

That’s why real apps use databases.

# 4. ID Generator

Your function:

```javascript
function generateId() {
  return crypto.randomBytes(12).toString('hex');
}
```

### What it does

Creates a **random unique identifier**.

---

### Pseudo Code

```text
function generateId:

    generate 12 random bytes

    convert to hex string

    return id
```

---

### Example Output

```text
generateId()

→ "7f9c2b4a6d8e1f3c9a0b12de"
```

Each file will get a **unique ID**.

# 5. Practical Example — File Upload

User uploads:

```text
profile.png
```

### Step 1 — Request comes to server

```text
Browser
   ↓
POST /upload
```

---

### Step 2 — Multer processes file

Example file object:

```javascript
file = {
  originalname: "profile.png",
  mimetype: "image/png",
  size: 203423
}
```

---

### Step 3 — Generate ID

```javascript
const id = generateId();
```

Example result:

```text
id = "f12a4b8c9e0d1f23a456789b"
```

---

### Step 4 — Create metadata object

Server creates:

```javascript
const metadata = {
  id: "f12a4b8c9e0d1f23a456789b",
  originalName: "profile.png",
  mimeType: "image/png",
  size: 203423,
  storage: "local",
  pathOrKey: "uploads/1710001-abc.png",
  uploadedAt: new Date()
};
```

---

### Step 5 — Store in memory database

```javascript
uploadedFiles.push(metadata);
```

Now the array becomes:

```javascript
uploadedFiles = [
 {
   id: "f12a4b8c9e0d1f23a456789b",
   originalName: "profile.png",
   mimeType: "image/png",
   size: 203423,
   storage: "local",
   pathOrKey: "uploads/1710001-abc.png",
   uploadedAt: "2026-03-09T22:10:00"
 }
]
```

# 8. Why This Pattern Is Useful

Even though this is temporary, it teaches an important backend pattern:

```text
File Storage  →  Metadata Storage
```

Example real system:

```text
File → AWS S3
Metadata → MongoDB
```

Instagram, Twitter, and GitHub all follow this pattern.

# 10. One Powerful Backend Habit

Whenever a system stores files, always separate:

```text
Binary Data → Object Storage (S3)
Metadata → Database
```

This makes the system **scalable and fast**.
