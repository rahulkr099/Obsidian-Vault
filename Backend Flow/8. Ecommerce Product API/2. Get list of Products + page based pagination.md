This part of the API is the **main product listing endpoint** of an ecommerce system.  
It handles many advanced features together:

- filtering
    
- sorting
    
- search
    
- field selection
    
- pagination (page or cursor)
    
- analytics (facets / price buckets)
    

Think of it like the **engine behind an Amazon-style product listing page**.

---

# 1. Big Idea

Client sends a request like:

```http
GET /products?category=phone&q=samsung&sort=price,-rating&page=2&limit=10
```

Your backend then:

1️⃣ reads query parameters  
2️⃣ builds filter conditions  
3️⃣ applies search  
4️⃣ builds sorting object  
5️⃣ selects fields if requested  
6️⃣ prepares pagination logic

Finally it queries MongoDB and returns products.

# 4. Query Parameters Extraction

Your code reads values from the URL.

Example request:

```http
GET /products?page=2&limit=10&q=samsung&sort=price,-rating&fields=name,price
```

Express parses it:

```javascript
req.query = {
 page: "2",
 limit: "10",
 q: "samsung",
 sort: "price,-rating",
 fields: "name,price"
}
```

---

# 5. Limit Validation

Your code:

```javascript
const limitNum = Math.min(100, Number(limit) || 20);
```

Pseudo logic:

```text
Convert limit to number

IF limit > 100
    set limit = 100

IF limit missing
    set limit = 20
```

Example:

```text
limit=500
```

Result:

```text
limitNum = 100
```

This prevents someone from requesting **10000 products at once**.

# 6. Build Filter Object

Your code calls:

```javascript
const filter = buildFilter(req.query);
```

Example request:

```http
GET /products?category=phone&minPrice=20000&inStock=true
```

Filter becomes:

```javascript
{
 category: "phone",
 stock: { $gt: 0 },
 price: { $gte: 20000 }
}
```

This filter will be passed to MongoDB.

# 7. Field Selection (Projection)

Your code:

```javascript
if (fields) projection = fields.split(',').join(' ');
```

Example request:

```http
GET /products?fields=name,price,rating
```

Projection becomes:

```javascript
"name price rating"
```

MongoDB query:

```javascript
Product.find(filter).select("name price rating")
```

Meaning:

```text
Return only these fields
```

This reduces API payload size.

# 8. Full Text Search

Your code:

```javascript
if (q) {
 filter.$text = { $search: q };
}
```

Example request:

```http
GET /products?q=samsung
```

Filter becomes:

```javascript
{
 $text: { $search: "samsung" }
}
```

MongoDB will search across indexed text fields.

Example matching products:

```text
Samsung Galaxy S23
Samsung Galaxy A54
Samsung Charger
```

# 10. Example Full Request Flow

### Request

```http
GET /products?q=samsung&category=phone&sort=price,-rating&fields=name,price&page=1&limit=5
```

---

### Step 1 — Query Params

```javascript
{
 q: "samsung",
 category: "phone",
 sort: "price,-rating",
 fields: "name,price",
 page: "1",
 limit: "5"
}
```

---

### Step 2 — Build Filter

```javascript
{
 category: "phone",
 $text: { $search: "samsung" }
}
```

---

### Step 3 — Projection

```javascript
"name price"
```

---

### Step 4 — Sort

```javascript
{
 price: 1,
 rating: -1
}
```

---

### Step 5 — Pagination

If page = 1 and limit = 5:

```text
skip = (page - 1) * limit
skip = 0
```

MongoDB query:

```javascript
Product.find(filter)
.select("name price")
.sort({ price: 1, rating: -1 })
.skip(0)
.limit(5)
```

# 14. Small Improvement Idea

A very powerful improvement is **query caching**.

For popular queries like:

```http
GET /products?category=phone
```

You can cache the response in **Redis**.

Flow becomes:

```text
Client
 │
 ▼
API Server
 │
 ▼
Check Redis Cache
 │
 ├── Cache Hit → return cached data
 │
 └── Cache Miss → query MongoDB
```

This can reduce database load **10x**.
