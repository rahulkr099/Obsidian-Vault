# What is Backend? 
Backend is a computer which is listening for HTTP or websocket or Grpc or any other kind of request through an open port which is accessible over the Internet. So that, the clients or other frontends can connect to it, send data to it, receive data from it.

## What is server?
A server is a computer or software system that provides data, services, or resources to other computers (clients) over a network. 

We call it server because it provides or it serves some kind of content whether it is static files like images, JS files, HTML files or it could be JSON and it also accept data if the client sends something. 

# The Flow of Request  👍  
Understanding this flow will make you a much stronger back-end developer (especially as you're working with MERN).

Let’s break it step-by-step with a **real example**.

---

# 🌍 Example Scenario

Suppose a user opens:

```
https://shortlyapp.in/login
```

(Just like your own project domain 😉)

Now let’s see how this request travels.

---

## 1️⃣ Browser → DNS Server

### What happens?

Your browser does NOT know where `shortlyapp.in` lives.

It asks DNS:

> “What is the IP address of shortlyapp.in?”

DNS responds:

```
shortlyapp.in → 3.110.25.189
```

So now the browser knows where to send the request.

📌 Think of DNS like your phone contacts app.

---

## 2️⃣ Browser → AWS Server (Public IP)

Now the browser sends an HTTP request to:

```
https://3.110.25.189/login
```

This IP belongs to your AWS EC2 instance.

Example AWS server:  Amazon Web Services

---

## 3️⃣ Firewall (Security Layer)

Before the request reaches your EC2 instance, it passes through:

- AWS Security Group
    
- Network ACL
    
- Possibly OS-level firewall (iptables)
    

### Example rule:

```
Allow inbound:
Port 80 (HTTP)
Port 443 (HTTPS)
```

If port 443 is closed → ❌ request blocked  
If allowed → ✅ request continues

Think of firewall as a **security guard at the gate**.

---

## 4️⃣ Request Reaches AWS EC2 Instance

Now the request reaches your virtual server.

Example:  Amazon EC2

But your Node.js app is NOT directly exposed.

Instead, Nginx is running first.

---

## 5️⃣ Nginx (Reverse Proxy)

Example:  
NGINX

Nginx listens on:

```
Port 80
Port 443
```

Your Node.js app runs on:

```
localhost:3000
```

### Nginx config example:

```nginx
server {
    listen 80;
    server_name shortlyapp.in;

    location / {
        proxy_pass http://localhost:3000;
    }
}
```

What happens now?

Nginx receives:

```
GET /login
```

Then forwards internally to:

```
http://localhost:3000/login
```

---

## 6️⃣ Node.js Server (Final Destination)

Your Express server (MERN backend) receives:

```js
app.get("/login", (req, res) => {
  res.send("Login Page");
});
```

Now your server processes:

- Authentication
    
- Database query
    
- JWT creation
    
- Response
    

Then sends response back.

---

# 🔁 Response Travels Back

Very important 👇

The response goes BACK the same way:

```
Node.js → Nginx → AWS EC2 → Firewall → Internet → Browser
```

Finally browser renders:

```
Login Page
```

Instagram

---

# 🚀 Step 1: You Click the Like Button

Inside the browser or mobile app:

JavaScript runs something like:

```js
POST /api/posts/789/like
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
```

This request contains:

- Post ID (789)
    
- Your User ID (from token)
    
- Timestamp
    

The app immediately shows ❤️ (optimistic UI)  
It does NOT wait for server response.

This makes the app feel fast.

---

# 🌍 Step 2: Request Travels to Server

Flow looks like this:

```
Mobile App / Browser
        ↓
DNS
        ↓
Load Balancer
        ↓
Application Server
        ↓
Database
```

Big platforms don’t use a single server.  
They use multiple servers behind a load balancer.

---

# 🧠 Step 3: Backend Processes the Like

On server side (example using Node.js):

```js
app.post("/api/posts/:id/like", async (req, res) => {
   const postId = req.params.id;
   const userId = req.user.id;

   await Likes.create({ postId, userId });

   await Posts.updateOne(
      { _id: postId },
      { $inc: { likeCount: 1 } }
   );

   res.json({ success: true });
});
```

What happens here:

1. A record is added in `likes` table
    
2. Post like count increases
    
3. Response sent back
    

---

# 🔔 Step 4: Notification System Triggers

Now comes the interesting part.

After saving the like:

The backend creates a notification event.

Something like:

```json
{
  "type": "LIKE",
  "fromUser": "Rahul",
  "toUser": "YourFriend",
  "postId": 789
}
```

Instead of sending notification directly, large systems use:

👉 Message Queue (like Kafka / RabbitMQ)

Why?

Because notifications should NOT slow down the main request.

---

# ⚡ Step 5: Background Worker Handles Notification

A background service listens to the queue:

```
Like Event → Queue → Notification Service
```

The notification service:

1. Saves notification in database
    
2. Sends push notification
    

---

# 📲 Step 6: Your Friend Gets Push Notification

Instagram uses:

- WebSockets
    
- Or Push Notification Service (Firebase/APNs)
    

Example:

If your friend is online:  
→ Real-time WebSocket message

If offline:  
→ Push notification stored  
→ Delivered when they open app

You see:

> Rahul liked your post ❤️

# 🧠 First: What Is “Frontend” Actually?

Frontend = code running in:

- Browser (Chrome, Firefox)
    
- Mobile App (React Native, Flutter)
    

Important truth:

> Frontend code runs on the USER’s device.

That means…

⚠️ The user controls it.  
⚠️ The user can inspect it.  
⚠️ The user can modify it.  
⚠️ The user can fake requests.

This is the core problem.

---

# 🚨 1️⃣ Security (Biggest Reason)

Imagine if Instagram handled likes like this:

```js
likeCount++;
```

If logic is only in frontend:

User can open DevTools and run:

```js
likeCount = 9999999;
```

Now what? 😅

Frontend cannot be trusted.

Example platform:  
Instagram

If likes were handled only in frontend:

- Anyone could fake likes
    
- Anyone could send fake notifications
    
- Anyone could modify data
    

That’s chaos.

---

# 🔐 2️⃣ Database Cannot Be Exposed

Imagine you connect MongoDB directly from browser:

```js
mongodb://username:password@server
```

Now every user can see:

- DB password
    
- Full database structure
    
- Modify or delete data
    

Game over.

Database credentials must stay on server.

---

# 🌍 3️⃣ CORS Is Not Just Restriction — It’s Protection

CORS prevents:

```text
randomwebsite.com → sending requests to your API
```

Without backend control:

Any malicious website could:

- Trigger actions
    
- Steal data
    
- Perform unwanted operations
    

Frontend alone cannot enforce access control properly.

---

# ⚡ 4️⃣ Heavy Computation

Imagine:

- Generating feed ranking
    
- Running AI recommendation
    
- Processing video uploads
    
- Running ML models
    

Example company:  Meta Platforms

Do you think they run recommendation algorithms on your phone?

No.

They use massive servers.

Reasons:

- CPU power
    
- Memory
    
- GPU
    
- Distributed systems
    

Browser is limited.

---

# 🧩 5️⃣ Business Logic Must Be Protected

Suppose your app has:

- Subscription validation
    
- Payment verification
    
- Premium content access
    

If logic is in frontend:

User can modify:

```js
isPremium = true;
```

Now they access paid features for free.

Backend enforces real validation.

---

# 🔄 6️⃣ Data Consistency

Imagine 1 million users liking same post.

If frontend handles it:

Everyone calculates locally.

Result:

- Inconsistent counts
    
- Race conditions
    
- Conflicts
    

Backend ensures atomic updates.

---

# 🛡 7️⃣ Authentication & Authorization

Frontend only stores token.

Backend verifies:

- Is token valid?
    
- Is user banned?
    
- Does user own this resource?
    

You cannot trust frontend for permission checks.

---

# 📦 8️⃣ Source Code Exposure

Frontend code is public.

Open DevTools → Sources → You see everything.

Backend code:

- Hidden
    
- Protected
    
- Controlled
    

---

# 🏗 Real Architecture Separation

Modern apps follow:

```text
Frontend = UI + UX
Backend = Logic + Security + Data + Control
```

Frontend:

- Sends request
    
- Displays response
    

Backend:

- Validates
    
- Computes
    
- Stores
    
- Enforces rules
    

---

✅ Local state management  
✅ Animations  
✅ UX improvements

But NOT for critical logic.
